1. Pourquoi abandonner la table unique centralisée ?

Contention cachée sur les bus inter‑CPU (UPI / Infinity Fabric)
Dans la version 3.7, bien que les opérations soient non‑bloquantes (atomic_cmpxchg), toutes les requêtes se concentrent sur la même table mémoire. Sur une machine multi‑socket (EPYC, Xeon), l’accès à cette table génère des invalidations de lignes de cache (MESI) et des traversées du bus inter‑processeur. En isolant physiquement les tables par CPU (__percpu), chaque cœur traite sa propre partition en local, supprimant toute contention mémoire entre cœurs.

2. Implémentation : structures alignées et accès sécurisé

Déclaration de la table distribuée et protection contre la migration CPU
/* structure d’un slot (inchangée) */
struct slot {
    atomic64_t key;
    atomic64_t value;
    atomic_t state;
} __aligned(64);

/* table par CPU avec cache‑line padding (élimine false sharing) */
struct skernel_table {
    struct slot slots[MAX_SLOTS_PER_CPU];
    atomic_t used_count;
    atomic_t tombstone_count;
    char _pad[____cacheline_aligned_in_smp];
};

/* allocation centrale dans module_init */
static struct skernel_table __percpu *cpu_tables;

cpu_tables = alloc_percpu(struct skernel_table);
if (!cpu_tables) return -ENOMEM;

/* accès local : migration interdite */
struct skernel_table *local = get_cpu_ptr(cpu_tables);
/* … opération sur la table locale … */
put_cpu_ptr(cpu_tables);
⚠️ Important : get_cpu_ptr() désactive la préemption et lie le thread au CPU courant. La migration est bloquée pendant l’accès, garantissant une affinité parfaite.

3. Élimination du « cache‑line bouncing »

Chaque cœur écrit dans sa propre ligne de cache – plus d’invalidation MESI
Avec la table unique, un atomic_cmpxchg sur une ligne de cache la porte en « Modified » sur le cœur écrivain et invalide toutes les copies des autres cœurs. Avec __percpu, chaque modification reste locale à la ligne de cache du cœur concerné. Le protocole MESI ne sollicite plus le bus inter‑socket. La scalabilité devient linéaire.

4. API utilisateur – routage par affinité

Comment l’appel système ioctl trouve‑t‑il la bonne table ?
À chaque requête ioctl (insertion, recherche, suppression), le module identifie le CPU appelant (via get_cpu_ptr) et redirige l’opération vers la table associée à ce CPU. Aucune logique de hachage répartie ni d’équilibrage n’est nécessaire : le noyau utilise naturellement la localité des threads de l’espace utilisateur.

5. Consolidation des résultats (phase MAP/REDUCE)

Comment obtenir une vue globale sans contention ?
Les métriques globales (ex. nombre total d’insertions) sont calculées uniquement lors de l’arrêt du module ou à la demande explicite via /proc/s_kernel_sentinel. Le module itère alors séquentiellement sur chaque table __percpu et somme les compteurs locaux. Pendant cette phase, aucune nouvelle insertion n’est bloquée – la cohérence finale est garantie par la conception monotone des compteurs.

6. Benchmarks prévus (validation expérimentale)

Métriques de performance pour comparer v3.7 vs v3.8
  • Latence p99 / p99.9 (insertion & recherche) – mesurée avec ktime_get_ns().
  • Cache misses (perf stat -e cache-misses,cache-references).
  • Scalabilité : montée en charge sur 2, 8, 32, 64 cœurs (EPYC / Xeon).
  • Contention mémoire – suivi des cycles UPI/Infinity Fabric via les compteurs PMU.
Les résultats bruts et les graphiques seront publiés dans un rapport de validation séparé (annexe technique).

7. Limites explicites et travaux futurs

Ce que cette version ne couvre pas (assumé)
  • Hotplug CPU : non géré – le module doit être rechargé après changement de topologie.
  • RCU / epoch reclamation : hors scope pour une preuve de concept déterministe.
  • Clustering multi‑machine : le sharding est intra‑serveur (une seule instance du module).
Ces aspects pourront être ajoutés dans des versions ultérieures si l’architecture est validée sur des clusters massifs.
